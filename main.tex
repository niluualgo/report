\chapter*{Abstract}

In high-density urban environments, losing an item is often a chaotic experience defined by panic and a lack of actionable data. Current methods rely on an "illusion of recovery," where people falsely believe lost items are easily found through traditional means, despite evidence to the contrary. 

This project proposes \textbf{"Reunite,"} a mobile application that transforms the lost-and-found process from a passive "waiting game" into an active, algorithmically secured recovery mission. The key innovation of this system is the implementation of a "Trust Network" which prevents scams and ensures secure handovers through asymmetric privacy and verification protocols. By utilizing geospatial scoring, real-time data synchronization, and automated greedy clustering for map visualization, the application streamlines recovery while protecting user identity and property.

% CHAPTER 1
\chapter{INTRODUCTION}

\section{Background}
The project addresses the experience of losing items in busy urban settings. This experience is specifically defined by panic and a complete lack of actionable data to track the item. Society currently suffers from an "illusion of recovery," convincing ourselves that lost items will be found, despite vast evidence to the contrary. In reality, we rely on outdated, traditional methods to find these items, which are often ineffective, disjointed, and insecure.

\section{Objectives}
The primary objective of this project is to create an active, algorithmically secured recovery platform. By replacing vague physical searching with dynamic spatial mapping and enforcing asymmetric privacy during handovers, the system aims to establish a secure community-driven "Trust Network" that dramatically increases the recovery rate of lost urban property.

\section{Problem Statement}
The traditional ecosystem of item recovery presents severe functional flaws. Scammers frequently use publicly posted photos of found items to fake ownership ("The Claim Scam"). Vague descriptions leave searchers with "Spatial Blindness," and overlapping digital pins create map clutter. Furthermore, old data stays up indefinitely, creating "Ghost Data" that reduces system efficiency.

\section{Proposed System}
To counter the limitations of the existing systems, the "Reunite" platform introduces advanced mechanisms including Asymmetric Privacy (hiding crucial details so prospective owners must prove knowledge), an Algorithmic Scoring Engine, $O(N)$ Greedy Clustering for maps, and Cryptographic QR Verification for secure handovers.

% CHAPTER 3
\chapter{PROJECT DESCRIPTION}

\section{Problem Definition}
The fundamental problem is the lack of a centralized, secure, and geospatially aware system to match lost and found items. The project aims to eliminate the friction, fraud, and inefficiency inherent in manual item recovery efforts.

\section{Overview}
The Reunite app is a modern, cross-platform mobile solution that utilizes real-time backend synchronization, advanced mathematical scoring for item matching, and interactive clustering maps to securely guide users through the item recovery process.

\section{System Architecture}
The system architecture operates across two primary layers: The Client Layer (built with Flutter and Mapbox for UI and spatial rendering) and the Backend Layer (utilizing Supabase and PostgreSQL for WebSockets and data management).

\section{Phases}

\subsection{Phase 1: Requirement Analysis and Foundation}
Identifying core user pain points, establishing hardware/software limitations, and designing the basic database schema required to handle geospatial coordinates.

\subsection{Phase 2: System Design and Security Planning}
Designing the UI/UX flows, integrating the Mapbox mapping system, and architecting the Asymmetric Privacy logic to prevent fraudulent claims.

\subsection{Phase 3: Module Development}
Building the core Flutter views, input forms, user authentication, and establishing the Supabase backend connection.

\subsection{Phase 4: AI and Scoring Integration}
Developing and integrating the mathematical scoring algorithm ($mScore$) that evaluates proximity, category matching, and tag similarity in real-time.

\subsection{Phase 5: Testing and Evaluation}
Conducting unit testing on the map clustering algorithms to ensure performance on low-end devices and testing the secure QR code handshake.

\subsection{Phase 6: Deployment and UI Polish}
Finalizing user interfaces, refining map visualization colors, preparing the application for deployment, and documenting the system performance.

\section{Overall Algorithm}
The system relies on an efficient scoring algorithm that mathematically evaluates multiple vectors: $S_{geo}$ (Geospatial Distance), $V_{time}$ (Freshness of the post), and $S_{attr}$ (Attribute overlap like categories and tags) to generate a match probability.

% CHAPTER 5
\chapter{SOFTWARE DESCRIPTION}

\section{Dart Programming Language}
Dart is an object-oriented, class-based language developed by Google. It is used as the primary coding language for the Reunite application due to its native performance capabilities and seamless integration with the Flutter framework.

\section{Flutter Framework}
Flutter is an open-source UI software development kit. It allows the system to compile natively to both mobile operating systems (Android and iOS) from a single unified codebase, ensuring a consistent user experience.

\section{Geospatial and Mapping Libraries}
The project integrates Mapbox, a powerful open-source mapping platform, to render custom, interactive maps. It utilizes specialized client-side libraries to handle spatial clustering dynamically based on the user's zoom level.

\section{Backend Technologies}
Supabase is utilized as the open-source Firebase alternative. It provides the application with secure edge functions and WebSocket integration, pushing real-time state updates to all users without manual refreshing.

\section{Database Management System}
The application relies on PostgreSQL, an advanced, enterprise-class open-source relational database. Paired with PostGIS, it efficiently handles the complex geospatial queries required to calculate distances between lost items.

\section{Development Tools}
Standard development environments such as Android Studio, VS Code, and Git were utilized for writing, testing, and version-controlling the application code.

Calculates the physical distance between reported locations.
\begin{lstlisting}[language=Java, caption=Proximity Score Engine]
double sGeo = (1 - (distance / 0.05)).clamp(0.0, 1.0);

\section{Tag Similarity Matching Algorithm}
Checks for overlaps in user-provided categories and text strings.
\begin{lstlisting}[language=Java, caption=Attribute Matching]
bool categoryMatch = targetLost['category'] != null && 
                     targetLost['category'] == found['category'];

String lostTags = (targetLost['tags'] ?? "").toString().toLowerCase();
String foundTags = (found['tags'] ?? "").toString().toLowerCase();

double sAttr = (lostTags.isNotEmpty && 
                foundTags.contains(lostTags.split(',').first.trim())) 
                ? 1.0 : 0.0;
\end{lstlisting}

\section{Final mScore Calculation}
\begin{lstlisting}[language=Java, caption=mScore Evaluation]
double mScore = (sGeo * 0.4) + (categoryMatch ? 0.4 : 0.0) + (sAttr * 0.2);
if (!categoryMatch) mScore = 0.05;
\end{lstlisting}

\chapter{ADVANTAGES AND DISADVANTAGES}

\section{Advantages}
The "Reunite" application offers several distinct advantages:
\begin{itemize}
    \item \textbf{Scam Immunity:} The Asymmetric Privacy engine eliminates visual bait, meaning scammers cannot use public photos to lay claim to items they do not own.
    \item \textbf{UX Clarity:} The integrated Greedy Clustering ensures maps remain highly usable, legible, and uncluttered.
    \item \textbf{High Performance:} The use of lightweight algorithms ensures the app remains snappy on older mobile devices.
    \item \textbf{Real-Time Agility:} WebSockets deliver millisecond updates, effectively eliminating "Ghost Data."
\end{itemize}

\section{Disadvantages}
Despite its significant improvements over legacy systems, the platform faces a few limitations:
\begin{itemize}
    \item \textbf{Dependency on User Base:} The effectiveness heavily relies on a critical mass of active users in the local area.
    \item \textbf{Hardware Constraints:} Users require consistent access to a smartphone with stable cellular data and active GPS hardware.
    \item \textbf{Indoor Geolocation Inaccuracy:} Standard GPS modules can struggle with pinpoint accuracy inside large, multi-story structures.
\end{itemize}
